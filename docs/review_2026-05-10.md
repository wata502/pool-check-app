## 1. 安定動作・バグ調査

### 1-1. `_safe_com_cleanup` の盲点（重大）

**現状の挙動（pool_writer.py L391–440）:**

```python
quit_ok = False
try:
    xl_app.Quit()
    quit_ok = True       # ← 例外なく戻れば成功扱い
except Exception:
    pass
...
if (not quit_ok) and pid:
    _kill_pid_force(pid)
```

**問題:** `Quit()` が **COM 例外を投げずに即座に return** したのに、内部で WindowsHook / マクロが残っていて EXCEL.EXE プロセスが残るケース（=「Quit 半成功」状態）は kill されない。Windows のタスクマネージャでだけ見える"幽霊プロセス"。これが累積すると、

- 共有ロックファイル `~$xxx.xlsx` を握ったまま残る → 以降の書き込みが永続的に "LOCKED" になる（最大5回バックオフ後 error）
- プロセス1つあたり 30〜80MB のRAMを保持 → 1日100件×残存5%＝400MB漏出を起こしうる

**修正案（差分）:**

```python
def _safe_com_cleanup(wb, xl_app):
    pid = _excel_pid(xl_app)
    # ... (既存処理: wb.Close, Quit) ...

    # ── ステップ 4 (改): Quit の成功/失敗にかかわらず、PID 生存を実検証 ──
    try:
        del wb
    except Exception:
        pass
    try:
        del xl_app
    except Exception:
        pass

    # GC を明示誘発（pywin32 の COM 参照は GC 待ちで解放される）
    import gc
    gc.collect()

    # PID がまだ生きていれば kill。Quit() が True を返していても実プロセス残存を検出する
    if pid and _pid_alive(pid):
        # Quit 後 1.0s 猶予を与え、EXCEL.EXE が自己終了するのを待つ（正常時は 100ms 以内に消える）
        for _ in range(10):
            if not _pid_alive(pid):
                break
            time.sleep(0.1)
        if _pid_alive(pid):
            logging.warning(
                f"[COM] EXCEL.EXE PID {pid} が Quit 後も生存 — taskkill /F で強制終了"
            )
            _kill_pid_force(pid)


def _pid_alive(pid):
    """OpenProcess で PID 生存を確認（CPU軽量）"""
    if not pid or win32process is None:
        return False
    try:
        import win32api, win32con
        h = win32api.OpenProcess(win32con.PROCESS_QUERY_LIMITED_INFORMATION, False, pid)
        if h:
            win32api.CloseHandle(h)
            return True
    except Exception:
        pass
    return False
```

**メリット:** Quit 半成功・サイレントゾンビ・COMリース不解放 を同じ経路でカバー。
**デメリット:** `gc.collect()` で 5〜30ms のスパイク。バッチ処理化（後述）と併用すれば回数自体が減るのでトレードオフ可。

### 1-2. `pythoncom.CoUninitialize()` の対称性違反

`write_pool_record` の `finally` で `CoUninitialize()` を**無条件で呼んでいる**。`CoInitialize()` が `CoInitializeEx` 既存呼び出し（GUIスレッド由来）と競合した場合 `CoInitialize` は `S_FALSE` 相当で「不要」と返るので、その場合に `CoUninitialize` を呼ぶと**呼び出しカウントが負になり予期せぬ COM 状態**を引き起こす。

**修正案:**

```python
co_inited = False
try:
    pythoncom.CoInitialize()
    co_inited = True
except pywintypes.com_error:
    pass  # 既存スレッドで初期化済み

try:
    # ... 業務処理 ...
finally:
    if co_inited:   # ← 自分で初期化したときだけ Uninit する
        try:
            pythoncom.CoUninitialize()
        except Exception:
            pass
```

### 1-3. `_recently_written` のロック保護なし

L954, L1061-1064, L1112, L1204 で **`_queued_ids_lock` 保護なしの dict 操作**が並ぶ。CPython では GIL のおかげで `del / setitem` 単発はクラッシュしないが、

- `for k, v in _recently_written.items()` 中に他スレッドが `__setitem__` → `RuntimeError: dictionary changed size during iteration`
- ワーカーアイドルの stale クリーンアップと、`PoolPoller._poll_loop` 経由の書き込みが衝突する経路は実在する

26件連続テストで再現しなかったのはタイミング依存。

**修正案:** 専用 Lock を追加。

```python
_recently_written      = {}
_recently_written_lock = threading.Lock()

# 書き込み (L1204)
with _recently_written_lock:
    _recently_written[record_id] = {
        "updatedAt": live_updated_at,
        "written_at": time.time(),
    }

# クリーンアップ (L1060)
now = time.time()
with _recently_written_lock:
    stale = [k for k, v in _recently_written.items()
             if now - v["written_at"] >= _IDEMPOTENCY_SEC]
    for k in stale:
        _recently_written.pop(k, None)

# 参照 (L1111)
with _recently_written_lock:
    cached = _recently_written.get(record_id)
    cached = dict(cached) if cached else None  # ロック外で安全に使うコピー
```

### 1-5. Web 側 `localStorage` の限界

**実装の現状:**
- `STORAGE_KEY_RECORDS` に `inspections` 全件を JSON 化して保存（L2340）
- 容量・例外ハンドリングなし
- iOS Safari プライベートモード = `setItem` で `QuotaExceededError` 投げる

**現実的なクラッシュシナリオ:**

1. 1件 ~5KB（特記事項+多数の測定値）× 事業所100件 = ~500KB → 余裕。
2. ただし**画像・サムネイル系**（将来追加された場合）や、`history` を切らずに1年運用すると 5MB 上限に達する端末が出る。
3. クォータ超過時、現状は**例外が握り潰されず投げ上がり**、保存処理自体が中断する → メモリ上の `inspections` だけが新しく、再起動でロールバックする「**サイレントロールバック**」リスク。

**修正案:**

```js
function saveInspections() {
  try {
    localStorage.setItem(STORAGE_KEY_RECORDS, JSON.stringify(inspections));
  } catch (e) {
    if (e && (e.name === 'QuotaExceededError' || e.code === 22)) {
      // 古い success 済みレコードを切り捨ててリトライ（直近90日のみ残す）
      const cutoff = Date.now() - 90 * 86400 * 1000;
      const trimmed = inspections.filter(r => {
        if (r.status !== 'success') return true;          // 未完了は残す
        if (!r.created_at) return true;
        return new Date(r.created_at).getTime() >= cutoff;
      });
      try {
        localStorage.setItem(STORAGE_KEY_RECORDS, JSON.stringify(trimmed));
        inspections = trimmed;
        console.warn('[Storage] quota exceeded — 90日超の success 済みレコードを切り捨て');
        showToast('容量逼迫のため古い書込済データを整理しました');
      } catch (e2) {
        console.error('[Storage] 切り捨て後も保存失敗:', e2);
        showToast('⚠ 端末の保存容量が不足しています', 'error');
      }
    } else {
      console.error('[Storage] 保存失敗:', e);
    }
  }
}
```

**より堅牢なアプローチ:** `IndexedDB` 移行。10MB→数GB に上限拡大、非同期 I/O でメインスレッド非ブロッキング。ただし全置換は大きな改修なので、まずは上記 quota ハンドリングで暫定対応を推奨。

## 2. Firebase 通信の堅牢性

### 2-1. SSE `auth_revoked` 未処理（重大）

**問題:** `PoolSSEListener._run_loop` (L1438-) は `event.event not in ("put", "patch")` で他のすべてのイベントを捨てる。Firebase RTDB は **`auth_revoked`** イベントを auth トークン期限切れ時に明示的に送るが、これを受信しても何もしない。

結果:
- トークン1時間期限切れ → Firebase が `auth_revoked` 送信 → ストリーム静かに無効化
- `READ_TIMEOUT=120s` で初めて気づく
- その間、SSE 接続は「生きているが空」状態 → SSE 経路で来るはずの新規レコードを **120秒見落とす**（ポーリングが救済するが POLL_INTERVAL=5s で運用的にもったいない）

**修正案:**

```python
for event in client.events():
    if self._stop_event.is_set():
        break

    # ── auth_revoked: 即時切断・トークン強制再取得 → ループの先頭で再接続 ──
    if event.event == "auth_revoked":
        self._log("[SSE] auth_revoked 受信 — トークンを再取得して再接続")
        with _fb_auth_lock:
            global _fb_auth_token, _fb_auth_expiry
            _fb_auth_token  = None
            _fb_auth_expiry = 0.0
        break  # 外側の while で再接続が走る

    # ── keep-alive / cancel は静かに継続 ──
    if event.event in ("keep-alive", "cancel"):
        continue

    if event.event not in ("put", "patch"):
        continue
    ...
```

**現状（L1425-1471）:**

`_queued_ids` set があるのでキュー重複は防げているが、**ネットワーク往復2回**を毎回踏んでいる。

**修正案:** どちらか一方に統一。SSE のフルデータが必ず最初に届くため、明示 `firebase_get_pending` は不要。

```python

# self._log(f"[SSE] 接続復帰 — 未処理レコードを確認中...")
```

**メリット:** 接続あたりの REST リクエスト1本削減、初回処理の遅延短縮。
**デメリット:** SSE フルデータが何らかの理由で届かない実装上のバグがあった場合の保険を失う → 心配なら **タイマー（10秒後に1回だけ）** で `firebase_get_pending` を呼ぶ救済路を残してもよい。

### 2-3. 複数デバイス同時アクセスでのデータ競合

**シナリオ:**
- 端末Aが `school_id=X / date=2026-05-10` の記録を Firebase に push（record_id_A）
- ほぼ同時に端末Bが同じ `school_id=X / date=2026-05-10` を push（record_id_B）
- PoolWriter は2件を順に処理 → Excel ファイル `X.xlsx` に**両方が書き込まれる**（後勝ち）。記録時刻のミリ秒差で結果が変動。

**実損:** 同一日付セルに端末B（後勝ち）の値が残るが、利用者は端末A視点で「自分の値が消えた」現象を見る。

**対策案（3パターン比較）:**

| アプローチ | 実装コスト | 整合性 | デメリット |
|---|---|---|---|
| **(A)** Firebase 側で `(school_id + date)` のキー重複時に PATCH ではなく既存 record_id へマージする | 中 | ◎ | クライアント側ロジック改修が必要 |
| **(B)** PoolWriter 側で同一 `(school_id, date)` を直近 10秒以内に検知したら `skip_reason="duplicate"` で第2件目を捨てる | 低 | ○ | 後発の正当な訂正を捨てるリスク（10秒バッファで運用上ほぼ無問題） |
| **(C)** Firebase Transaction で原子的に "1分子1記録" を保証 | 高 | ◎ | Realtime Database の transaction はリトライ多発でレイテンシ増 |

**推奨は (B) を即時導入、長期で (A) へ。コード例:**

```python
# pool_writer.py の _process_one 内、Excel 書込み直前に挿入
_school_recent_lock = threading.Lock()
_school_recent: dict = {}   # {(school_id, date_str): (record_id, ts)}

def _is_school_duplicate(school_id, date_str, record_id):
    """直近 10 秒で同じ (school_id, date) が既に書き込まれていれば True"""
    if not school_id or not date_str:
        return False
    key = (school_id, date_str)
    now = time.time()
    with _school_recent_lock:
        prev = _school_recent.get(key)
        if prev and (now - prev[1]) < 10 and prev[0] != record_id:
            return True
        _school_recent[key] = (record_id, now)
    return False

# ...呼び出し側:
header = record.get("header") or {}
if _is_school_duplicate(school_id, header.get("date", ""), record_id):
    self._log(f"  ⏭ 直近重複検出 — スキップ: {record_id}")
    firebase_patch_status(firebase_url, record_id,
        {"status": "skipped", "skip_reason": "duplicate_within_10s"}, api_key=api_key)
    self._skip_count += 1
    return
```

### 2-4. トークン期限切れ判定の Race

**L156:** `if _fb_auth_token and now < _fb_auth_expiry - 300:` のチェックを通った直後にトークンが本当に切れる稀ケース（時計ドリフト or サーバ側強制失効）。`_http_request` の 401 リトライ路があるので致命傷ではないが、**`_fb_auth_url(url, api_key)` が SSE 用 GET URL に焼き込まれた古いトークン**は再リフレッシュで救えない（既に send 済みの URL）。

**SSE の場合の対策:** `auth_revoked` 経路（2-1）と組み合わせて、ストリーム接続前に強制的にトークン年齢をチェック。

## 3. パフォーマンス（バッチ処理化）

ここが今回最大の改善余地。実測ベース（一般的な win32com Excel 起動時間）で見積もる。

### 3-1. 現状の所要時間内訳（1件あたり）

| 処理              | 想定時間          | 備考                                      |
|-------------------|-------------------|-------------------------------------------|
| `DispatchEx`      | 1.0〜2.0 秒        | Excel.exe プロセス起動                     |
| `Workbooks.Open`  | 0.8〜2.5 秒        | ファイルサイズ・マクロ有無に依存             |
| データ書込        | 0.1〜0.4 秒        | 50〜100セル更新                              |
| 特記事項 Justify  | 0.2〜0.5 秒        | 結合セル時は手動ラップで軽い                 |
| `Save`            | 0.5〜1.5 秒        | 再計算込み                                 |
| `Close + Quit`    | 0.4〜1.0 秒        |                                           |
| **合計**          | **3.0〜7.9 秒**    |                                           |

**同一ファイル N 件を連続処理した場合:**
- 現状: `N × 5秒` = 線形増加
- バッチ化（同一ファイル再オープン省略）: `N × 1.0秒 + 起動5秒` 程度 ≈ **60〜80% 短縮**

26件のテストでは school_id がバラついていれば効果は薄いが、**1学校の点検データを同時送信する運用**では決定的な差になる。

### 3-2. 実装アプローチ（3案）

#### (A) ピーク・バッチ方式（推奨：実装コスト低）

ワーカーが1件取り出した時点で、`queue.get_nowait()` を最大 N 回繰り返してキューを peek し、**同じ excel_path のレコード**だけ集約。残りはキュー先頭に戻す。

```python
def _drain_same_target(self, first_record_id, max_batch=20):
    """
    キューから同じ excel_path のレコードだけ吸い上げて返す。
    異なるレコードは drain 中に出現したら即時にキュー先頭に戻す（順序保持）。
    """
    first_record = self._fetch_record(first_record_id)
    if not first_record:
        return [(first_record_id, None)]

    school_id = first_record.get("school_id", "")
    school_info = self.mapping_data.get(school_id) or {}
    file_name = school_info.get("file_name", "")
    if not file_name:
        return [(first_record_id, first_record)]

    excel_path = find_excel_file(self.cfg.get("excel_folder", ""), file_name)
    batch = [(first_record_id, first_record)]
    deferred = []

    while len(batch) < max_batch:
        try:
            rid = _task_queue.get_nowait()
        except queue.Empty:
            break
        if rid is None:
            _task_queue.put(None)   # 停止シグナルは戻す
            break
        rec = self._fetch_record(rid)
        if not rec:
            _task_queue.task_done()
            continue
        ssid = rec.get("school_id", "")
        ssinfo = self.mapping_data.get(ssid) or {}
        sfile = ssinfo.get("file_name", "")
        spath = (find_excel_file(self.cfg.get("excel_folder", ""), sfile)
                 if sfile else None)
        if spath == excel_path:
            batch.append((rid, rec))
            # task_done() はバッチ処理後に呼ぶため、ここでは呼ばない
            # ただし _queued_ids からの除去は処理完了後（既存仕様）
        else:
            deferred.append(rid)
            _task_queue.task_done()    # peek した分は明示で done

    # 順序保持で deferred を戻す
    for rid in deferred:
        _task_queue.put(rid)
    return batch, excel_path, school_info


def _process_batch(self, batch, excel_path, school_info):
    """1つの Excel を開いて batch 全件を書き込み、Save→Close→Quit は1回だけ"""
    if not WIN32_AVAILABLE:
        return
    pythoncom.CoInitialize()
    xl_app = wb = None
    try:

        xl_app.Visible = False
        xl_app.DisplayAlerts = False
        xl_app.ScreenUpdating = False
        xl_app.EnableEvents = False
        wb = xl_app.Workbooks.Open(excel_path)
        xl_app.Calculation = -4135  # Manual

        for record_id, record in batch:
            try:
                _write_pool_record_into_open_wb(wb, xl_app, record, school_info)
                # 各レコードの冪等性キャッシュ更新は呼び出し側で
                self._mark_success(record_id, record)
            except Exception as e:
                logging.error(f"[Batch] {record_id} 書込失敗: {e}")
                self._mark_error(record_id, str(e))

        xl_app.Calculation = -4105
        wb.Save()
        wb.Saved = True
        wb.Close()
        xl_app.Quit()
        wb = xl_app = None
    except Exception:
        _safe_com_cleanup(wb, xl_app)
    finally:
        try:
            pythoncom.CoUninitialize()
        except Exception:
            pass
```

`write_pool_record` 本体を「Excel 起動／クローズを含まない `_write_pool_record_into_open_wb(wb, xl_app, record, school_info)` 」と「外殻の起動/クローズ」に分離するリファクタが必要。**既存ロジックの責務分離だけ**で破壊変更は無い（既存テスト通過のはず）。

| 観点         | 評価                                                                  |
|--------------|-----------------------------------------------------------------------|
| 実装コスト   | 低（既存関数のリファクタのみ）                                         |
| 速度効果     | 同一ファイル連続時 60〜80% 短縮                                        |
| 安定性       | 1件失敗時の挙動を「他はSave、自分だけerror」と決めれば従来同等         |
| 副作用       | バッチ途中でロックされた場合に全件 LOCKED 扱いになる→部分リトライ要      |
| 影響最小化   | 単発処理パスは温存（`max_batch=1` で完全に従来動作に戻せる）           |

#### (B) 永続 Excel インスタンス方式（最大速だが複雑）

ワーカーが Excel.Application を起動しっぱなしにし、**ファイルを開閉するだけ**で `Quit` は終了時のみ。

| 観点         | 評価                                                                  |
|--------------|-----------------------------------------------------------------------|
| 実装コスト   | 中（インスタンス所有・健康診断・自動再起動のロジックが必要）            |
| 速度効果     | 同一ファイル: (A) と同等 / 異なるファイル: 起動2秒削減＝1件あたり10〜30%短縮 |
| 安定性       | **長期生存 Excel は COM 状態破損のリスク累積**（メモリリーク・GDIハンドル枯渇） |
| 副作用       | 利用者が手動で別の Excel を開いた場合の競合シナリオが増える              |
| 影響最小化   | 失敗時の影響範囲が大きい（全 pending が止まる）                         |

**注意:** 週1回ペースで内部 Application を強制再起動する自衛が必要。本格運用に入るまでは(A)を推奨。

#### (C) `xlwings` / `openpyxl` への一部移行

`win32com` はマクロ実行や複雑書式（Justify）が必要な場合に限定し、純粋なセル値書込みは **`openpyxl`** で代替。

| 観点         | 評価                                                                  |
|--------------|-----------------------------------------------------------------------|
| 実装コスト   | 高（Justify 等価機能を自前で実装する必要あり）                         |
| 速度効果     | Excel 起動不要 → 1件 0.5〜1.0秒/件                                     |
| 安定性       | プロセス独立性が完全担保（zombie が原理的にゼロ）                       |
| 副作用       | xlsm マクロ・条件付き書式の一部・特殊書式が壊れる可能性                  |
| 影響最小化   | テンプレート互換性の徹底検証が必要                                     |

**現状の `Justify` 依存と xlsm 運用を考えると、(C) は中長期の選択肢。短期は(A)。**

### 3-3. その他の細かい高速化

1. **`xl_app.Calculation = -4135 (Manual)` の効果が活きていない**: 現状でも書いているが、`Save()` 直前に `-4105` に戻すので結局保存時に再計算が走る。**書式が複雑なファイル**では `Save()` 前の `Calculate()` を明示し、Save 中の自動再計算を抑える方が速い場合がある。
2. **`_set_cell` が `ws.Cells(row, col)` を毎回呼ぶ**: COM 通信が1セル1往復。100セルで100往復。範囲書込で `ws.Range("D5:D14").Value = ((v1,),(v2,),...)` の2D配列 1往復に集約できる。50ms × 50回節約 = 2.5秒/件。
3. **`pool_mapping.json` 49KB を起動時1回読むだけ**: 問題なし。
4. **`time.sleep(1.0)` 削除済（L880 コメント）**: Good。

| 優先度 | 工数 | 項目                                                          |
|:--:|:--:|:------------------------------------------------------------|
| ★1 | 1d | `_safe_com_cleanup` の PID 生存検証・`gc.collect()` 追加（1-1） |
| ★1 | 1d | SSE `auth_revoked` ハンドリング追加（2-1）                      |
| ★1 | 3d | バッチ処理化アプローチ(A)（3-2A）                                |
| ★2 | 0.5d | `_recently_written` のロック追加（1-3）                        |
| ★2 | 0.5d | localStorage quota ハンドリング（1-5）                          |
| ★2 | 1d | 重複 `(school_id, date)` の10秒スキップ（2-3B）                |
| ★3 | 0.5d | `CoUninitialize` 対称性修正（1-2）                            |
| ★3 | 0.5d | `_EXCEL_PATH_CACHE` LRU 化（1-4）                            |
| ★3 | 0.5d | SSE 接続復帰時の二重投入削除（2-2）                              |

26件連続成功は良いベースラインだが、上記★1の3項目は**運用 100件/日を超えるとほぼ確実に**観測される。先に着手すること。
