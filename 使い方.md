# プール付属電気設備点検システム 使い方

## 構成

```
プール点検/
├── pool_writer.py         ← PC側 Excel書き込みツール（ソース）
├── PoolWriter.exe         ← exe（build_exe.bat でビルド後）
├── pool_mapping.json      ← 学校別マッピング情報
├── setup_mapping.py       ← マッピング自動生成スクリプト（初回のみ）
├── build_exe.bat          ← exe ビルド
├── スマホアプリ更新.bat   ← Firebase デプロイ
├── マッピング再生成.bat   ← setup_mapping.py 実行
├── firebase.json          ← Firebase Hosting 設定
└── public/                ← Firebase Hosting 公開フォルダ
    ├── index.html         ← スマホ入力アプリ（本体。編集対象はこのファイル）
    ├── manifest.json
    ├── sw.js
    └── weather_judge.js
```

---

## セットアップ手順

### Step 1: マッピング生成（初回のみ）

1. Excel ファイル（.xlsm）が格納されているフォルダを確認する
2. `マッピング再生成.bat` をダブルクリック
3. フォルダパスを入力（例: `C:\Users\wata5\Dropbox\プール点検\2026年度`）
4. `pool_mapping.json` が生成される
5. 内容を確認し、学校名・ファイル名・行番号が正しいか確認する

### Step 2: スマホアプリをデプロイ

1. `スマホアプリ更新.bat` をダブルクリック
2. Firebase Hosting にアップロードされる
3. スマホのブラウザで開き「ホーム画面に追加」する

### Step 3: PC ツールをセットアップ

1. `build_exe.bat` をダブルクリックして `PoolWriter.exe` をビルド
2. `PoolWriter.exe` をダブルクリックして起動
3. 「設定」タブで以下を入力:
   - Firebase URL（例: `https://inspection-data-74326-default-rtdb.firebaseio.com`）
   - Web API キー
   - メールアドレス / パスワード
   - **Excel フォルダ**（プール点検の xlsm ファイルがあるフォルダ）
4. 「保存して接続テスト」をクリック
5. 「監視開始」をクリック

## 運用フロー

```
【現場】
  ① スマホでアプリを開く
  ② 学校をタップ
  ③ 基本情報（日付・天候・温度・湿度）を入力
  ④ 絶縁抵抗・接地抵抗・漏電遮断器の測定値と判定を入力
  ⑤ 特記事項があれば入力（定型文選択も可）
  ⑥ [一括送信] ボタンをタップ

【PC側（自動）】
  ⑦ PoolWriter.exe が Firebase を検知
  ⑧ pool_mapping.json を参照してファイルを特定
  ⑨ Excel を開いて各セルに値を書き込み
  ⑩ 保存してログに記録
```

## よくある問題

| 症状 | 対処法 |
|------|--------|
| スマホアプリに学校一覧が出ない | Firebase 設定を確認。マッピングが未設定の場合は内蔵データで表示される |
| 「マッピングが見つかりません」エラー | setup_mapping.py を実行して pool_mapping.json を再生成 |
| Excel ファイルが見つからない | PoolWriter の「Excel フォルダ」設定を確認 |
| 「Excelが開かれています」エラー | 対象ファイルを閉じてから自動リトライされる |
| 特記事項が割り付けられない | Justify() の実行に失敗した場合は A39 にそのまま書き込まれる |

## デプロイしてもスマホアプリに反映されない時の対処

### 原因

`スマホアプリ更新.bat`（=`firebase deploy --only hosting`）で配信ファイルは更新されているが、ブラウザ側で **Service Worker が旧キャッシュを返している** ために、最新の `pool_mapping.json` や `index.html` が見えていない状態。Ctrl+Shift+R（スーパーリロード）でも、Service Worker のキャッシュは消えないことがある。

### 対処（必ずデプロイ前に実施）

`public/sw.js` の **11 行目** にある `APP_VERSION` を毎回書き換える。

```js
const APP_VERSION = 'v11-20260502-03';   // ← ここを書き換える
```

書き換え後の例（命名規則: `vNN-YYYYMMDD-NN`）:

```js
const APP_VERSION = 'v12-20260508-01';
```

`APP_VERSION` が変わると `CACHE_NAME` も変わるため、新 SW の activate 時に**旧キャッシュが全削除**される（`sw.js` L65-82 の activate ハンドラ）。1バイトでも変えればよい。

### 反映までの正しい手順

1. `public/sw.js` 11 行目の `APP_VERSION` を新しい文字列に書き換えて保存
2. `スマホアプリ更新.bat` を実行（Deploy Success! を確認）
3. PC Chrome では `Ctrl+Shift+R` でリロード
4. スマホ PWA は次のアプリ起動時に自動更新。即時反映したい時は **設定 → アプリ情報 → ストレージ削除** で強制クリア

### それでも反映されない時の最終手段

PC Chrome で `F12` → Application タブ → Service Workers → `Unregister` → ページを再読込。スマホ側はアプリのアンインストール → 再インストール。

### 確認方法

PC Chrome の DevTools コンソールで以下のいずれかが出ているか確認:

- `[Mapping] Firebase から取得: NN 校` … Realtime Database 経由で取得（pool_mapping ノード参照）
- `[Mapping] ローカルファイルから取得: NN 校` … Hosting 上の `pool_mapping.json` を取得

「ローカルファイルから取得」と出ているのに古いままなら、Service Worker キャッシュの可能性がほぼ確定。上記手順で `APP_VERSION` を上げ直す。

## 注意事項

- `pool_mapping.json` は **setup_mapping.py で自動生成** が推奨。手動編集する場合はJSONの構文に注意。
- プール運用なし（`has_data: false`）の学校はスマホアプリでグレー表示され選択不可。
- 220_田尻高島交流館は対象外。
