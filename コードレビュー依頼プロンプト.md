## レビュー観点

### 1. バグ・潜在的な不具合

- 未定義変数、型の不一致、nullチェック漏れ
- エラーハンドリングの不足（try-catch / try-except）
- 条件分岐の漏れ、境界値の考慮不足
- API呼び出し（天気APIなど）の失敗時・タイムアウト時の処理
- 非同期処理（async/await、Promise）の例外捕捉漏れ
- ブラウザ間の互換性問題（特にiOS Safari / Android Chrome）
- データの整合性（pool_mapping.jsonの構造とコード側の参照が一致しているか）

### 2. 不要コード（安全に削除可能なもの）

- 使われていない関数・変数・import・CSS
- コメントアウトされたまま残っている旧コード
- デバッグ用の console.log / print文
- 到達不能なコード（dead code）
- 重複している処理（同じロジックが複数箇所にあるもの）

**判断基準：**
- 「将来使う可能性がある」ものは残す判断でOK
- 明らかに不要なものだけ指摘すること
- 判断が微妙なものは「要確認」として別途記載

### 3. パフォーマンス改善（動作を軽くする）

- DOM操作の効率化（不要な再描画、重いループ内でのDOM操作）
- 大きなJSONデータ（pool_mapping.json: 約44KB）の読み込み・処理の最適化
- イベントリスナーの重複登録がないか
- メモリリーク（解放されない参照、蓄積されるデータ）
- Service Workerのキャッシュ戦略の妥当性（キャッシュ肥大化のリスク）
- 画像ファイルのサイズ・形式の最適化余地（現在 icon-512.png が約300KB）
- CSS/JSの不要な肥大化がないか

### 4. セキュリティ

- APIキーやシークレットのハードコードがないか
- XSS（クロスサイトスクリプティング）の脆弱性
  - innerHTMLへの未サニタイズなデータ挿入
  - ユーザー入力値のエスケープ処理
- 外部APIとの通信がHTTPSで行われているか
- Firebase設定の公開範囲が適切か
- Service Workerのスコープ設定が適切か

### 5. PWA・オフライン対応の完全性

- オフライン時に必要なリソースがすべてキャッシュ対象になっているか
- キャッシュの更新戦略（古いキャッシュが残り続けないか）
- manifest.jsonの設定が適切か（アイコン、テーマカラー、表示モード）
- 「ホーム画面に追加」後の動作に問題がないか

### 6. 安全性の確認（既存動作を絶対に壊さないこと）

- 各指摘について「修正した場合の影響範囲」を必ず明記すること
- 修正リスクを3段階で評価すること
  - **低リスク**：単純な削除・修正で他に影響なし
  - **中リスク**：関連箇所の同時修正が必要
  - **要注意**：動作に影響する可能性があり、十分なテストが必要
- 他機能への副作用がないか確認すること
- 「修正しない方がよい」場合はその旨を明記すること

## レポート形式

### ファイルごとのセクション

各指摘には以下を記載してください：

- **該当箇所**：ファイル名・行番号（または行の範囲）
- **カテゴリ**：バグ / 不要コード / パフォーマンス / セキュリティ / PWA
- **問題の内容**：何が問題なのか具体的に説明
- **推奨する対応**：どう修正すべきか（コード例があれば尚良い）
- **修正リスク**：低 / 中 / 要注意
- **優先度**：高 / 中 / 低

### サマリー（レポート末尾）

1. **優先度「高」の項目一覧**（すぐに対応すべきもの）
2. **優先度「中」の項目一覧**（時間があるときに対応）
3. **優先度「低」の項目一覧**（余裕があれば対応）
4. **全体の所見**（アプリ全体の品質に対する総合コメント）
